TOPOLOGY & ROUTING
El sistema está diseñado bajo un patrón de Microservicios. Todos los servicios independientes están registrados en un servidor Eureka (Discovery Server). Sin embargo, por seguridad táctica, el cliente (Postman o Navegador) NUNCA debe comunicarse directamente con los microservicios individuales.

Toda la comunicación externa debe pasar a través del API Gateway (Puerto 8080), el cual actúa como el único punto de entrada (Single Point of Entry) y enruta el tráfico internamente.

LAYERED DESIGN (CAPAS)
Internamente, cada microservicio opera mediante una arquitectura de capas estricta para asegurar la separación de responsabilidades (Separation of Concerns).
1. CONTROLLER LAYER
Maneja las peticiones HTTP (GET, POST). Expone los Endpoints y recibe los DTOs. Pasa la orden al Service.
2. SERVICE LAYER
Contiene el "Business Logic" (Lógica de Negocio). Realiza cálculos, valida reglas del juego y se comunica con otros microservicios si es necesario.
3. REPOSITORY LAYER
Interactúa directamente con la Base de Datos mediante Spring Data JPA.

INTER-SERVICE COMMUNICATIONS
¿Cómo hablan los microservicios entre sí si están separados? Utilizan OpenFeign (FeignClients).

Cuando un servicio necesita datos de otro (Por ejemplo, el Combate-Service necesita saber el daño de un arma alojada en Arma-Service), el Combate-Service realiza una llamada REST interna a través de Feign, de forma síncrona.
// Ejemplo de FeignClient en Inventario-Service pidiendo datos a Arma-Service @FeignClient(name = "arma-service", path = "/api/armas") public interface ArmaClient { @GetMapping("/{id}") ArmaDTO obtenerArmaPorId(@PathVariable("id") Long id); }

POSTMAN TACTICAL SETUP
Para ejecutar pruebas eficientes en Postman durante la presentación al cliente:
  1. Crea una Colección llamada "VideoJuego Online - EXAMEN TRANSVERSAL".
  2. Configura una Variable de Entorno (Environment): {{baseUrl}} = http://localhost:8080.
  3. Usa la variable en todas tus rutas: {{baseUrl}}/api/armas. Si el Gateway cambia de puerto, solo cambias la variable.
  4. Recuerda enviar el Body en formato raw -> JSON para las peticiones POST y PUT.
STREAMING ARCHITECTURE
EXAMEN TRANSVERSAL — Pipeline de Streaming en Tiempo Real

A diferencia de la arquitectura Batch de EV2, este proyecto responde a un escenario donde los eventos de negocio (compra/venta de componentes en la subasta Duoc UC) se generan de forma continua e impredecible, exigiendo una arquitectura capaz de ingerir, desacoplar y procesar la informacion a medida que ocurre.

El pipeline se construyo integramente sobre Google Cloud Platform (GCP), con los siguientes componentes:
DUOC UC WEBHOOK
CLOUD RUNPython/Flask
PUB/SUBTOPIC_VENTAS
DATAFLOWApache Beam
CLOUD STORAGEData Lake
ALTERYX / DATAPREP
LOOKER STUDIO
Componentes clave del pipeline streaming:
  • Cloud Run — Microservicio REST en Python/Flask que recibe el webhook y publica en Pub/Sub.
  • Pub/Sub — Bus de mensajeria asincrona (TOPIC_VENTAS, suscripcion ventas-sub).
  • Dataflow (Apache Beam) — Motor de procesamiento en streaming con ventanas de 12 segundos.
  • Cloud Storage — Data Lake con politica de retencion TTL de 30 dias.
  • Alteryx / Cloud Dataprep — Preparacion y limpieza de datos (datos_limpios_nombrados).
  • Looker Studio — Visualizacion final con dashboard interactivo.
BATCH VS STREAMING
Arquitectura Batch (EV2): Procesamiento historico de archivos Parquet con Apache Spark. Los datos se procesan en lotes programados (horarios fijos). Latencia de horas o dias. Ideal para analisis retrospectivos y generacion de reportes periodicos.

Arquitectura Streaming (EV3): Procesamiento en tiempo real via webhook. Los eventos se capturan, publican y procesan en ventanas de 12 segundos. Latencia de extremo a extremo de minutos. Ideal para monitoreo en vivo y toma de decisiones operativas.
BATCHEV2 — Spark
Archivo Parquet → Spark → Alteryx → Looker
Latencia: horas / dias
STREAMINGEV3 — Dataflow
Webhook → Cloud Run → Pub/Sub → Dataflow → Storage
Latencia: minutos
HERRAMIENTAS DE IMPLEMENTACION
Las herramientas que materializan esta arquitectura se operan desde tres entornos principales:

Google Cloud Console: Interfaz web de GCP para gestionar Cloud Run, Pub/Sub, Dataflow, Cloud Storage e IAM. Permite monitorear el estado de cada servicio en tiempo real, revisar logs y configurar alertas.

PowerShell / Terminal (gcloud CLI): Interfaz de linea de comandos para automatizar el despliegue mediante scripts. Comandos como gcloud run deploy, gcloud pubsub topics create y gcloud dataflow jobs run ejecutan todo el pipeline de forma reproducible.

Navegador web: Verificacion del webhook via POST requests, validacion de endpoints REST en Cloud Run y visualizacion del dashboard final en Looker Studio. Tambien se utiliza para autenticacion OAuth y configuracion de la API de Duoc UC.

Este stack de herramientas cubre desde la configuracion inicial hasta la validacion final del pipeline, permitiendo tanto la automatizacion por script como la administracion visual via consola web.
