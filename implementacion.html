PIPELINE STREAMING — GCP DEPLOY
Operacion: Pipeline Ventas Tiempo Real — DuocUC → GCP

Arquitectura: DuocUC (POST Webhook) → Cloud Run → Pub/Sub → Dataflow (Apache Beam) → Cloud Storage (CSV) → Alteryx/Cloud Dataprep → Looker Studio

El despliegue del pipeline involucra la configuracion secuencial de cada componente GCP, desde la ingesta de datos via microservicio REST hasta la visualizacion final en dashboard.

PASO 0 — DETECTAR REGION
Antes de iniciar el despliegue, es necesario identificar la region de GCP permitida por las politicas del proyecto. La politica de ubicacion de recursos (gcp.resourceLocations) define en que regiones se pueden crear los servicios. Dependiendo del proyecto, la region habilitada puede ser us-east1, us-east4, us-central1 o europe-west1. Esta region debe configurarse de forma consistente en todos los servicios del pipeline para evitar errores de comunicacion entre componentes.
PASO 0.1 — VARIABLES DE ENTORNO
Se definen variables de entorno que identifican los recursos del pipeline: el ID del proyecto GCP (PROJECT_ID), el nombre del topico Pub/Sub (TOPIC_VENTAS), el nombre del servicio Cloud Run (ventas-api), la imagen Docker (ventas-api:v1) y el bucket de Cloud Storage cuyo nombre incorpora el PROJECT_ID para garantizar unicidad global. Estas variables se utilizan en todos los pasos posteriores del despliegue.
PASO 0.2 — CONFIGURAR PUB/SUB
Se crea un topico Pub/Sub llamado TOPIC_VENTAS que actua como bus de mensajeria asincrona entre el microservicio de ingesta y el pipeline de procesamiento. Adicionalmente se crea una suscripcion ventas-sub con tipo PULL y un ack-deadline de 60 segundos, que permite a Dataflow consumir los mensajes de forma controlada. Pub/Sub garantiza la entrega al menos una vez (at-least-once) y desacopla la produccion del consumo de eventos.
PASO 1 — MICROSERVICIO CLOUD RUN
1.1 — Aplicacion Python/Flask: Se desarrolla un microservicio REST en Python utilizando Flask que expone un endpoint POST en el puerto 8080. El servicio recibe el payload JSON del webhook de Duoc UC, extrae los 10 campos del schema (id_cliente, cliente, genero, id_producto, producto, precio, cantidad, monto, forma_pago, fecreg), los serializa a formato CSV y los publica al topico Pub/Sub. Incluye manejo de errores para excepciones GoogleAPICallError y RetryError, y soporta tanto objetos individuales como arreglos de ventas.

1.2 — Dockerfile: Se crea una imagen Docker basada en python:3.11-slim que copia la aplicacion, instala las dependencias (Flask, google-cloud-pubsub) y expone el puerto 8080.

1.3 — Build, Push y Deploy: La imagen se construye localmente con Docker, se etiqueta para Container Registry de GCP (gcr.io/{PROJECT_ID}/ventas-api:v1) y se sube al registro. Luego se despliega en Cloud Run como servicio serverless con 512MB de memoria, timeout de 300 segundos y las variables de entorno PROJECT_ID y TOPIC_ID configuradas. El servicio se despliega con autenticacion deshabilitada para recibir webhooks externos.

La URL generada por Cloud Run (ej: https://ventas-api-{PROJECT_ID}.{REGION}.run.app) se registra en la plataforma Duoc UC para que los webhooks de la subasta sean dirigidos al pipeline.
PASO 2 — CLOUD STORAGE
Se crea un bucket de Cloud Storage con nombre unico global (ventas-{PROJECT_ID}-df) en la misma region configurada. Este bucket funciona como Data Lake del pipeline, almacenando los archivos CSV generados por Dataflow. El bucket se estructura en tres carpetas: temp/ para archivos temporales del procesamiento, staging/ para datos en estado intermedio y samples/output/ para los CSV finales listos para consumo por las herramientas de visualizacion. Se aplica una politica de retencion TTL de 30 dias para control de costos.
PASOS 3-4 — ROLES Y PRIVILEGIOS IAM
Para que los servicios de GCP puedan comunicarse entre si, se configuran los permisos IAM necesarios. Se habilitan las APIs requeridas: Dataflow, Compute Engine, Pub/Sub, Cloud Storage, IAM, Cloud Resource Manager y Cloud Run.

La cuenta de servicio por defecto de Compute Engine ({PROJECT_NUMBER}-compute@developer.gserviceaccount.com) recibe los roles necesarios para operar el pipeline:
  • roles/dataflow.worker — Permite ejecutar workers de Dataflow
  • roles/storage.objectAdmin — Permite leer y escribir objetos en Cloud Storage
  • roles/pubsub.editor — Permite publicar y consumir mensajes de Pub/Sub

Adicionalmente, el agente de servicio de Dataflow (service-{PROJECT_NUMBER}@dataflow-service-producerprod.iam.gserviceaccount.com) requiere el rol roles/dataflow.serviceAgent y permisos de usuario sobre la cuenta de servicio worker para ejecutar el pipeline.
PASOS 5-6 — DATAFLOW PIPELINE
El pipeline de procesamiento en streaming se construye con Apache Beam SDK 2.49.0 en Java, utilizando el codigo de referencia de Google Cloud Platform. El pipeline se configura con los siguientes parametros:

  • Entrada: Suscripcion Pub/Sub ventas-sub
  • Salida: Bucket Cloud Storage en gs://{BUCKET_NAME}/samples/output
  • Ventana: Fija de 12 segundos para agrupar mensajes
  • Runner: DataflowRunner en modo streaming
  • Autoescalado: Deshabilitado, con 1 worker fijo (n1-standard-1)
  • Ubicacion temporal: gs://{BUCKET_NAME}/temp

El pipeline lee mensajes desde Pub/Sub via PubsubIO, los agrupa en ventanas de 12 segundos y escribe archivos CSV particionados en Cloud Storage. El proceso de compilacion y ejecucion se realiza con Maven, y el job puede monitorearse en tiempo real desde la consola de Dataflow, que muestra metricas de throughput, watermark y utilizacion de workers.
PASOS 7-8 — VERIFICACION Y RESULTADOS
Una vez desplegado el pipeline completo, se verifican los siguientes puntos:

  • Cloud Run: El microservicio responde a peticiones HTTP GET/POST en la URL asignada. Se puede verificar con una peticion curl o directamente desde el navegador.
  • Pub/Sub: Los mensajes publicados por Cloud Run aparecen en la suscripcion ventas-sub. Se pueden inspeccionar desde la consola de GCP o mediante comandos de verificacion.
  • Dataflow: El pipeline debe mostrar estado RUNNING en la consola de Dataflow, con datos de throughput y ventanas de 12 segundos.
  • Cloud Storage: Despues de ~20 minutos de ejecucion, aparecen archivos CSV en gs://{BUCKET_NAME}/samples/output/ con el formato: id_cliente, cliente, genero, id_producto, producto, precio, cantidad, monto, forma_pago, fecreg

MANEJO DE ERRORES
Durante el despliegue pueden presentarse los siguientes escenarios:

Error de region en Dataflow: Si la region configurada no esta dentro de las permitidas por la politica del proyecto, el pipeline falla. Se debe verificar la politica de ubicacion de recursos y reconfigurar la region en todos los servicios.

Bucket duplicado: Si el bucket ventas-{PROJECT_ID}-df ya existe (por ejemplo, de un intento anterior), se puede crear uno nuevo con un sufijo alternativo como -df2.

Cloud Scheduler: Para pruebas automatizadas, se puede configurar un job de Cloud Scheduler que publique mensajes de prueba en el topico Pub/Sub cada minuto. Esto permite verificar el funcionamiento del pipeline sin depender del webhook externo de Duoc UC.

Errores comunes adicionales:
  • 403 Forbidden: Verificar permisos IAM. La cuenta de servicio de Dataflow necesita roles dataflow.worker, pubsub.subscriber y storage.objectAdmin.
  • Pub/Sub push timeout: El webhook en Cloud Run debe responder en menos de 60 segundos. Verificar que el handler no haga procesamiento sincrono pesado.
  • Dataflow no escribe outputs: Verificar que el schema del mensaje Pub/Sub coincida con lo esperado por el pipeline Beam.
  • Looker Studio sin datos: Verificar que el archivo CSV en Cloud Storage tenga los nombres de columna correctos y que la conexion este activa.

Stack de monitoreo: Cloud Logging para logs centralizados, Cloud Monitoring para alertas de latencia y errores, dashboards de Dataflow para throughput y backlog de Pub/Sub.
GCP CONSOLE WORKFLOW
Los mismos pasos del despliegue pueden realizarse desde la interfaz grafica de Google Cloud Console, complementando el enfoque de automatizacion:

1. Cloud Run (UI): Consola → Cloud Run → Crear servicio. Seleccionar la imagen del contenedor desde Artifact Registry, configurar puerto 8080, asignar variable de entorno TOPIC_ID y habilitar autenticacion. La URL del endpoint se genera automaticamente.

2. Pub/Sub (UI): Consola → Pub/Sub → Crear topico (TOPIC_VENTAS). Crear suscripcion (ventas-sub) con tipo PULL, ack-deadline de 60s y retencion de 7 dias. Desde la UI se puede publicar mensajes de prueba y visualizar el backlog.

3. Dataflow (UI): Consola → Dataflow → Crear trabajo desde plantilla o codigo propio. Seleccionar pipeline Beam, especificar suscripcion Pub/Sub de entrada y bucket de salida. La UI muestra graficos de throughput, watermark y utilizacion de workers en tiempo real.

4. Cloud Storage (UI): Consola → Cloud Storage → Navegador de buckets. Visualizar archivos CSV generados por Dataflow, configurar TTL de 30 dias y administrar permisos de acceso. La UI permite previsualizar archivos sin descargarlos.

5. IAM (UI): Consola → IAM y administracion → Asignar roles a cuentas de servicio. La cuenta de servicio de Dataflow requiere: dataflow.worker, pubsub.subscriber y storage.objectAdmin.
