RESUMEN EJECUTIVO
Este proyecto implementa una arquitectura Batch sobre Google Cloud Platform para el análisis masivo de datos históricos de la NYC TLC (Taxi & Limousine Commission). El pipeline automatizado ingesta archivos Parquet del repositorio oficial de NYC, los procesa por lotes en Cloud Run, los almacena en Cloud Storage como Data Lake, aplica limpieza ETL en Cloud Dataprep/Alteryx, los carga en BigQuery como Data Warehouse y finalmente los visualiza en Power BI.

Dataset: NYC Yellow Taxi Trip Data (Enero 2026) — 387.7 MB CSV, millones de registros
Stack: Cloud Scheduler → Pub/Sub → Cloud Run (Docker) → Cloud Storage → Dataprep/Alteryx → BigQuery → Power BI
Arquitectura: Batch (procesamiento por lotes programados)
FUNDAMENTOS BIG DATA — 5V
Volumen: El dataset base presenta un tamaño considerable de 387.7 MB en formato CSV, equivalente a millones de registros de viajes individuales. Este volumen excede la capacidad de herramientas ofimáticas tradicionales, justificando el uso de Cloud Storage y BigQuery.

Velocidad: Pipeline diseñado bajo un modelo de procesamiento por lotes (Batch). Cloud Scheduler orquesta la ingesta en intervalos programados, capturando y procesando el gran volumen de información de forma periódica.

Variedad: La información ingresa en formato Parquet y se materializa como CSV tabular, conteniendo tipos de datos heterogéneos: texto (identificadores de proveedor), variables temporales (fechas de inicio y fin del viaje) y numéricas (distancias, tarifas, propinas, recargos).

Veracidad: Para asegurar la calidad de los datos crudos, el proceso ETL aplica reglas estrictas de limpieza que mitigan inconsistencias como valores nulos, tarifas negativas y registros con distancia cero, garantizando que el análisis final se base en información fidedigna.

Valor: Transformar los datos crudos en insights estratégicos conectados a Power BI permite extraer conocimiento accionable: horarios de mayor congestión, zonas de alta demanda y promedios de ingresos para la toma de decisiones del negocio.

PASO 1 — CLOUD RUN
Se despliega un microservicio REST en Python/Flask que descarga el archivo Parquet del repositorio oficial de NYC TLC, lo procesa en lotes de 150,000 filas usando Apache Parquet y pandas, y transmite el resultado como CSV directamente a Cloud Storage vía streaming para evitar problemas de memoria (OOM). Adicionalmente publica metadatos del proceso al tópico Pub/Sub.
FILE main.py
Microservicio Flask que procesa Parquet por lotes y escribe CSV a GCS en streaming.
import os, io, json, logging, requests import pandas as pd import pyarrow.parquet as pq from datetime import datetime from flask import Flask, jsonify from google.cloud import storage app = Flask(__name__) DATA_URL = "https://d37ci6vzurychx.cloudfront.net/trip-data/yellow_tripdata_2026-01.parquet" OUTPUT_PREFIX = "tlc/trips" def download_parquet(): response = requests.get(DATA_URL, timeout=300) response.raise_for_status() return response.content def process_and_upload_parquet(content, csv_filename): blob = bucket.blob(csv_filename) pf = pq.ParquetFile(io.BytesIO(content)) with blob.open("wt", encoding="utf-8") as f: for i, batch in enumerate(pf.iter_batches(batch_size=150000)): df_chunk = batch.to_pandas() df_chunk.to_csv(f, index=False, header=(i == 0)) @app.route("/", methods=["POST", "GET"]) def main(): content = download_parquet() timestamp = datetime.now().strftime("%Y%m%d_%H%M%S") csv_filename = f"{OUTPUT_PREFIX}/yellow_tripdata_{timestamp}.csv" processed = process_and_upload_parquet(content, csv_filename) publish_to_pubsub(metadata) return jsonify({"status":"success","records_processed":processed["total_records"]})

PASO 3 — ETL: LIMPIEZA Y TRANSFORMACION
El núcleo del éxito analítico reside en la preparación de datos. El flujo ETL se estructura en tres ejes fundamentales, ejecutados en Cloud Dataprep y Alteryx Designer:
ETL 1. LIMPIEZA DE DATOS
Purga de transacciones inválidas: Eliminación sistemática de registros con trip_distance ≤ 0 o total_amount ≤ 0. Estos registros representan transacciones truncadas, viajes cancelados o fallas en la calibración del taxímetro.

Aislamiento de anomalías temporales: Filtro para descartar registros con fechas atípicas fuera del rango objetivo, originados por desconfiguraciones de hardware en los terminales de los vehículos.

Deduplicación: Validación a nivel de fila para identificar y remover registros exactos duplicados, evitando la inflación artificial de ingresos.
ETL 2. NORMALIZACION DE DATOS
Tratamiento de valores nulos: Estandarización de celdas vacías (NULL o NaN) en columnas críticas como payment_type y RatecodeID. Previene el quiebre de funciones de agregación (SUM, AVG) en consultas SQL masivas.

Casteo de tipos de datos (Typecasting): Forzado de la tipología correcta: identificadores como enteros (INTEGER), métricas de recaudo como decimales (FLOAT), campos cronológicos como DATETIME.
ETL 3. TRANSFORMACION Y ENRIQU
Métricas derivadas temporales: A partir del campo tpep_pickup_datetime se extraen atributos específicos: "Hora del día" y "Día de la semana". Esta transformación alimenta los gráficos de distribución temporal en el dashboard sin sobrecargar el motor de base de datos con cálculos en tiempo de ejecución.

Optimización del esquema: Descarte de columnas sin valor para los requerimientos del negocio. Esta reducción de dimensionalidad disminuye el peso del archivo procesado, optimizando la velocidad de transferencia y minimizando costos de almacenamiento.
PASO 4 — BIGQUERY

PASO 5 — POWER BI
La herramienta de visualización final. Power BI Desktop se conecta a BigQuery para tomar la información ordenada y transformarla en gráficos interactivos e informes visuales que ayudan a la toma de decisiones.

Los dashboards incluyen:
  • Distribución horaria: Volumen de viajes por hora del día, identificando horas punta y valles de demanda.
  • Zonas de alta demanda: Mapa de calor de ubicaciones de recogida y destino más frecuentes.
  • Análisis de tarifas: Promedio de tarifas por zona, tipo de pago y hora del día.
  • KPI operativos: Distancia promedio por viaje, ingresos por día, propinas por zona.
ROLES & GOBERNANZA DE DATOS
Gobierno de Datos: La prioridad es la Gestión de Calidad del Dato. Se implementan reglas de validación en Dataprep para limpiar inconsistencias, eliminar viajes con duración cero e imputar valores nulos en tarifas cobradas.

Ciclo de Vida del Dato:
  • Captura: Descarga mensual estructurada desde el repositorio oficial de NYC TLC.
  • Almacenamiento: Guardado inicial en el Data Lake (Cloud Storage).
  • Transformación: Limpieza y estandarización del formato de fechas, coordenadas y tarifas.
  • Archivado: Tras 5 años, los datos históricos pasan a Cold Storage para optimizar costos.

CONCLUSION
La implementación de este pipeline Batch en Google Cloud Platform demostró ser una solución altamente eficaz para el procesamiento masivo de registros históricos de transporte. A través de la automatización orquestada con Cloud Scheduler, Pub/Sub y Cloud Run (en contenedores Docker), se logró una ingesta robusta y escalable hacia el Data Lake en Cloud Storage.

El uso de Cloud Dataprep resultó vital para la fase de limpieza, permitiendo erradicar inconsistencias críticas del archivo original como tarifas negativas o registros duplicados que afectaban la veracidad de la información.

Al alojar datos normalizados en BigQuery, se facilitó la auditoría analítica mediante consultas SQL y se preparó el modelo para su conexión directa con Power BI, transformando un volumen masivo de datos crudos en conocimiento estratégico accionable para optimizar operaciones, comprender la demanda y maximizar la rentabilidad del negocio.
