SERVICE DISCOVERY — EUREKA
Netflix Eureka actúa como un "Directorio Telefónico" dinámico para los microservicios. Cuando un servicio (ej. arma-service) se inicia, se registra automáticamente en Eureka enviándole su nombre y puerto. Si el API Gateway necesita enrutar tráfico a arma-service, consulta a Eureka que le responde con la IP y puerto exactos del nodo vivo.

Load Balancing: Eureka permite múltiples instancias del mismo servicio. El API Gateway utiliza Spring Cloud LoadBalancer con algoritmo Round-Robin para distribuir la carga.
# application.yml eureka: client: service-url: defaultZone: http://localhost:8761/eureka/ instance: instance-id: ${spring.application.name}:${random.value}
CONFIG SERVER
Spring Cloud Config Server centraliza toda la configuracion de los microservicios en un solo repositorio. Cuando un servicio arranca, solicita su archivo .yml al Config Server antes de iniciar sus procesos. Esto evita tener que modificar cada servicio individualmente cuando cambia una contraseña o un puerto.
# application.yml (local) spring: application: name: personaje-service config: import: optional:configserver:http://localhost:8888/
FAULT RESILIENCE
Circuit Breaker: Si un servicio falla, las llamadas sincronicas (Feign) no deben quedar esperando indefinidamente. Se implementa un patron de resiliencia que corta el circuito y ejecuta un fallback controlado, evitando el "Efecto Domino" (Cascading Failure).

Global Exception Handler: @RestControllerAdvice captura todas las excepciones (FeignException, EntityNotFoundException, etc.) y las envuelve en un DTO de error estandarizado.
{ "timestamp": "2026-06-25T03:30:00Z", "status": 503, "error": "Service Unavailable", "message": "Arma-Service unreachable. Fallback executed.", "path": "/api/combates" }
SECURITY & CORS
Solo el API Gateway (Puerto 8080) esta expuesto al exterior. Funciona como firewall y checkpoint de seguridad, concentrando la configuracion de CORS y JWT. Los microservicios individuales no son accesibles directamente desde internet.
# API Gateway CORS Config spring: cloud: gateway: globalcors: cors-configurations: '[/**]': allowedOrigins: "*" allowedMethods: [GET, POST, PUT, DELETE]
STREAMING COMPONENTS — GCP
Componentes del pipeline de streaming (EXAMEN TRANSVERSAL):
API CLOUD RUN
Microservicio REST en Python/Flask que recibe el webhook de Duoc UC via POST y publica cada evento como mensaje en Pub/Sub. Incluye manejo de errores GoogleAPICallError y RetryError.
BUS PUB/SUB
Bus de mensajeria asincrona. Topico: TOPIC_VENTAS. Suscripcion: ventas-sub con ack-deadline de 60 segundos. Desacopla la ingesta del procesamiento.
PROC DATAFLOW (APACHE BEAM)
Motor de procesamiento en streaming. Pipeline Java que consume mensajes de Pub/Sub, los agrupa en ventanas fijas de 12 segundos y escribe CSV en Cloud Storage.
DLK CLOUD STORAGE
Data Lake con carpetas temp/, staging/ y samples/output/. Politica de retencion TTL de 30 dias para control de costos.
ETL ALTERYX / CLOUD DATAPREP
Motor visual de preparacion de datos. Toma los CSV crudos del Data Lake y aplica reglas de limpieza, generando el dataset datos_limpios_nombrados.
DASH LOOKER STUDIO
Visualizacion final. Tablero interactivo con metricas: distribucion por genero, volumen de transacciones por empresa y montos por producto.
DETALLE DE COMPONENTES
Cloud Run — receiver-subastas: Contenedor Python/Flask desplegado en europe-west1. Expone endpoint POST /webhook que valida el payload JSON, lo enriquece con timestamp y lo publica al topico Pub/Sub. Incluye manejo de reintentos para errores transitorios.

Pub/Sub — TOPIC_VENTAS: Topico creado con retencion de mensajes de 7 dias. Suscripcion ventas-sub con ack-deadline de 60 segundos y politica de reintentos. Garantiza entrega al menos una vez (at-least-once).

Dataflow — pipeline-beam: Pipeline en Java construido con Apache Beam SDK 2.49.0. Lee desde la suscripcion Pub/Sub con PubsubIO, aplica ventanas fijas de 12s y escribe CSV particionados al bucket de Data Lake. Configurado para autoescalado hasta 10 workers.

Cloud Storage — bucket-ev3: Data Lake en clase STANDARD con TTL de 30 dias. Tres directorios: temp/ (procesamiento intermedio), staging/ (datos sin procesar), samples/output/ (datos listos para consumo).

Alteryx: Flujo ETL que lee CSV desde Cloud Storage, estandariza nombres (datos_limpios_nombrados), aplica transformer de tipos. Dataprep: Alternativa nativa GCP para limpieza visual de datos, ideal como reemplazo de Alteryx en entornos cloud-native.

Looker Studio: Dashboard publicado con 8 graficos. Conexion directa a Cloud Storage. Dataset de ~5600 registros con campos: fecha, genero, empresa, producto, monto. Actualizacion cada 15 minutos mediante programacion.
FLUJO DE HERRAMIENTAS
Los componentes del pipeline se operan y verifican a traves de un flujo integrado de herramientas:

1. GCP Console (Google Cloud Console): Interfaz web principal para crear y gestionar todos los recursos. Desde la consola se despliega Cloud Run, se crean topicos Pub/Sub, se lanzan pipelines Dataflow, se administran buckets de Cloud Storage y se configuran permisos IAM. Ideal para administracion visual y monitoreo de logs.

2. PowerShell con gcloud CLI: Automatizacion de todo el ciclo de vida del pipeline mediante scripts. Los comandos gcloud permiten reproducir el despliegue de forma deterministica, facilitando la integracion continua y la documentacion ejecutable.

3. Alteryx / Cloud Dataprep: Herramientas de preparacion de datos. Alteryx ofrece un flujo ETL visual con nodos de transformacion, mientras que Dataprep es la alternativa nativa de GCP integrada con Cloud Storage. Ambas leen los CSV generados por Dataflow y producen datasets limpios.

4. Looker Studio: Plataforma de visualizacion que consume los datos limpios desde Cloud Storage. Los dashboards se actualizan cada 15 minutos mediante conexion programada.

5. Navegador web: Punto de verificacion final. Se utiliza para probar el webhook via POST requests, validar respuestas del microservicio Cloud Run y visualizar el dashboard publicado.
